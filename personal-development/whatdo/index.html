Quick Mode (Default)

When someone says "what should we do?" with no other context, don't ask questions ‚Äî just GO.

Quick Mode Logic

  1. Check the clock ‚Äî day of week, time of day
  2. Check the calendar ‚Äî any planned events today/tonight? Conflicts? (see Calendar Integration)
  3. Check the weather ‚Äî use web_search for current conditions at user's location (read location from USER.md)
  4. Check routines ‚Äî is it Taco Tuesday? First Friday? A seasonal tradition?
  5. Check history ‚Äî what have they done recently? What's overdue?
  6. Check preferences ‚Äî known favorites, group profiles, game library
  7. Generate ONE confident suggestion with the full output format

Examples:

  • Saturday night + nice weather + usually goes out ‚Üí "Hit up that new cocktail bar downtown ‚Äî patio weather!"
  • Tuesday night + rainy + usually stays in ‚Üí "You've got Catan and it's been a while ‚Äî game night?"
  • It's Tuesday ‚Üí "Taco Tuesday! Want the usual spot or mix it up?"
  • October + weekend ‚Üí "It's spooky season ‚Äî haunted house time?"
  • Haven't done an escape room in 3 months ‚Üí "You're overdue for an escape room ‚Äî there's a new one downtown"

If preferences are too thin to make a confident Quick Mode suggestion, fall back to the full question flow.

The Flow (Full Mode)

When Quick Mode doesn't have enough context, or the user wants to explore options, run through these questions. Keep it conversational and snappy ‚Äî this is NOT a survey. It's a fun back-and-forth. Use inline buttons when available, or quick-fire options.

If the platform supports inline buttons, present each question with tappable options. Otherwise, list them conversationally.

Live Weather Integration

Before generating suggestions, always check the weather at the user's location.

How to Check Weather

  1. Read USER.md for the user's current location
  2. Use web_search to search for current weather: "weather [city] today" or "current weather [city]"
  3. Parse the temperature, conditions (sunny/rainy/cloudy/etc.), and forecast

Weather Decision Logic

Condition Action
Clear/sunny, 60-85¬∞F Push outdoor options hard ‚Äî "Perfect night to be outside!"
Partly cloudy, mild Outdoor-leaning, mention "bring a layer"
Rainy/stormy Auto-pivot to indoor ‚Äî "Rain's coming down ‚Äî let's make it a cozy night"
Extreme heat (100¬∞F+) Indoor or water activities ‚Äî "It's scorching ‚Äî pool, AC, or wait for sunset"
Cold (<40¬∞F) Indoor or cold-weather fun ‚Äî "Bundle up for a bonfire or stay in with cocoa"
Snow Embrace it or hide from it ‚Äî "Fresh snow = sledding, or fire + hot cocoa"

Business Hours & Ratings

Business Hours

When suggesting places to go, always check if they're open.

With Google Places API:

  • Request the currentOpeningHours field in every query
  • Filter out closed businesses ‚Äî never suggest somewhere that's closed
  • Show hours in output: "Open until 11pm" or "Closes in 2 hours"
  • If closing soon (<1 hour), warn: "‚ö†Ô∏è Closes at 10pm ‚Äî hustle!"

Without Google Places API:

  • Add a note: "Check hours on Google Maps before heading out"
  • Use web_search as a fallback to find hours for specific venues

Ratings Floor

With Google Places API:

Favorites & Blacklist

How It Works

  • Favorites ‚Äî places and activities the user loves. Resurface them periodically:
    • "You loved Ichiban Ramen ‚Äî haven't been in a while!"
    • "You always have a great time at escape rooms ‚Äî there's a new one in town"
  • Blacklist ‚Äî places and activities to NEVER suggest:
    • Blacklisted places are invisible. Period. Don't mention them.
    • Disliked activities are filtered out entirely.
  • Building the lists ‚Äî after every suggestion, offer:
    • "üëçüëé How'd we do? (helps me learn your taste)"
    • Thumbs up ‚Üí ask if they want to add it to favorites

Checking Before Suggesting

Before presenting any suggestion:

  4. Check favorite_activities ‚Äî lean into known loves

With Google Calendar API

If GOOGLE_CALENDAR_API_KEY or Google Calendar OAuth credentials are available, create events via the API:

  -H "Authorization: Bearer $GOOGLE_CALENDAR_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "summary": "Game Night ‚Äî Catan Tournament üé≤",
    "location": "Scott'\''s RV",
    "description": "Game night with the crew. Bring beer (not Dave). Sarah gets veggie pizza.",
    "start": {

      "timeZone": "America/Phoenix"
    },
    "end": {
      "dateTime": "2026-01-28T23:00:00-07:00",
      "timeZone": "America/Phoenix"
    },
    "attendees": [
      {"email": "mike@example.com"},
      {"email": "sarah@example.com"}

Event creation details:

  • Summary: Activity name + emoji for fun
  • Location: From the suggestion or user-specified
  • Description: Include group context, what to bring, dietary notes
  • Attendees: Pull email addresses from group member contacts (only those with email field)
  • Reminders: Default to 2 hours and 30 minutes before
  • Duration: Default 4 hours for group activities, 3 hours for date night, 2 hours for casual
  • Store the returned event_id in history.json as calendar_event_id

Google Places Integration (Optional Enhancement)

If the environment variable GOOGLE_PLACES_API_KEY is available, use it to enhance suggestions with real, nearby places.

How to Use

Text Search (best for specific types):

curl -s -X POST 'https://places.googleapis.com/v1/places:searchText' \
  -H "Content-Type: application/json" \
  -H "X-Goog-Api-Key: $GOOGLE_PLACES_API_KEY" \
  -H "X-Goog-FieldMask: places.displayName,places.formattedAddress,places.rating,places.userRatingCount,places.priceLevel,places.googleMapsUri,places.types,places.currentOpeningHours" \
  -d '{
    "textQuery": "best ramen restaurant in Scottsdale AZ",
    "maxResultCount": 5
  }'

Nearby Search (best for "near me" suggestions):

curl -s -X POST 'https://places.googleapis.com/v1/places:searchNearby' \
  -H "Content-Type: application/json" \
  -H "X-Goog-Api-Key: $GOOGLE_PLACES_API_KEY" \
  -H "X-Goog-FieldMask: places.displayName,places.formattedAddress,places.rating,places.userRatingCount,places.priceLevel,places.googleMapsUri,places.types,places.currentOpeningHours" \
  -d '{
    "includedTypes": ["restaurant"],
    "maxResultCount": 5,
    "locationRestriction": {
      "circle": {
        "center": {"latitude": 33.8303, "longitude": -111.9258},
        "radius": 16000
      }
    }
  }'

Places Processing Rules

  1. Always request currentOpeningHours ‚Äî filter out places that are currently closed
  2. Apply ratings floor ‚Äî default 4.0 stars, or min_rating from preferences
  3. Sort by rating ‚Äî highest first
  4. Include userRatingCount ‚Äî show as "‚≠ê 4.6 (2,341 reviews)"
  5. Show hours ‚Äî "Open until 11pm" or calculate "Closes in 2 hours"
  6. If closing soon (<1 hour) ‚Äî add warning: "‚ö†Ô∏è Closes at 10pm ‚Äî hustle!"
  7. Always include googleMapsUri ‚Äî direct link for navigation
  8. Show price level ‚Äî translate to $ symbols

Without Places API

  • Still give amazing suggestions ‚Äî just describe the type of place
  • Add: "Search Google Maps for '[type of place] near me' to find the perfect spot"
  • Suggest checking hours: "Make sure to check hours on Google Maps before heading out"
  • The skill works great either way; Places just adds a cherry on top
